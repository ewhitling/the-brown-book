## ADR 1.0: Data extraction from epub
**Date:** 2026-04-26
**Status:** Accepted

**Context** — Need a stable text foundation from a single `source.epub` containing all five novels in three omnibus groupings (Shadow & Claw, Sword & Citadel, Urth).

**Decision** — Python script using `zipfile` + `beautifulsoup4` to extract directly from epub XHTML. Chapter-to-book mapping derived from file naming conventions (`b1-chapter1..35` = Shadow, `b1-chapter36..66` = Claw, etc.) and validated against the epub's TOC. Output as JSON with chapter-level granularity. Tetralogy in `data/books.json`, essays and Urth in `data/ignored/`.

**Consequences** — No dependency on epub-specific libraries. Extraction is idempotent and deterministic. Titles come from TOC (single source of truth) rather than per-chapter HTML parsing. Derived data is gitignored; `source.epub` is also gitignored (large binary). Regenerate with `make extract`.

## ADR 1.1: Three-tier epistemics for research questions
**Date:** 2026-04-26
**Status:** Accepted

**Context** — Research questions range from countable facts to literary interpretation. Risk of presenting interpretive claims as proven findings.

**Decision** — Every research question is tagged as one of:
- **Extractive** — answer is enumerable from text (vocab counts, death events, entity appearances)
- **Synthetic** — requires restructuring/summarization but is grounded in text (chapter summaries, timeline reconstruction)
- **Interpretive** — requires literary judgment; we provide evidence, not proof (story meanings, cyclical structure)

All outputs must clearly state which tier they occupy. Interpretive claims must cite specific textual evidence.

**Consequences** — Slower to produce results. Forces us to separate "what the text says" from "what we think it means." Prevents overclaiming.

## ADR 2.0: SQLite as canonical data store
**Date:** 2026-04-26
**Status:** Accepted

**Context** — JSON files require full-scan for cross-chapter queries (e.g., "all chapters where Dorcas appears"). Research questions #2 (contradictions), #5 (timeline), #6 (hidden characters) need relational joins. JSON intermediates cost ~$6 in API calls to regenerate.

**Decision** — Single SQLite database (`data/botns.db`, ~3MB) as the canonical store. 10 tables: `books`, `chapters` (text + summary in same row), `characters` (one row per appearance, `name` indexed for cross-chapter joins), `unnamed_characters`, `locations`, `temporal_markers`, `factual_claims`, `notable_details`, `embedded_narratives`, `embedded_narrative_characters`. Database is checked into git. JSON files remain as gitignored intermediates.

**Consequences** — All downstream analysis becomes SQL queries instead of Python loops. No API cost to access data in new sessions. Character deduplication deferred (name column is the join key for now, but "Dr. Talos" vs "Doctor Talos" variants exist). Regenerate with `make db`.

## ADR 2.1: Self-contained HTML summary browser
**Date:** 2026-04-26
**Status:** Accepted

**Context** — Need a human-readable interface to browse chapter summaries, characters, claims, and embedded narratives. Options: pure static HTML (no navigation state), Svelte (adds node/npm toolchain), single HTML with embedded JSON + vanilla JS.

**Decision** — Single self-contained HTML file (`data/browser.html`) generated by `build_browser.py`. Summary data (no full text) embedded as JSON. Vanilla JS for navigation. Zero external dependencies. "Dark Observatory" aesthetic (dark bg, indigo accent, Avenir Next).

**Consequences** — Opens with `file://`, can be emailed to collaborators. No build toolchain beyond Python. Limited interactivity compared to a framework, but sufficient for a research workbench. Checked into git alongside the database.

## ADR 3.0: Scene decomposition as a second API extraction layer
**Date:** 2026-04-26
**Status:** Accepted

**Context** — Chapter-level summaries (ADR 2.1) are too coarse for research questions about pacing, travel, and narrative mode (#5 timeline, journey reconstruction). A finer unit is needed without re-summarizing full chapters.

**Decision** — `extract_scenes.py` decomposes all 144 chapters into 853 scenes via the Claude API, capturing narrative mode (action, dialogue, reflection, embedded story, dream, description), location, characters present, and travel direction. Scenes are stored in the SQLite database (`scenes`, `scene_characters`) alongside chapter data. A separate local pass, `analyze_vocab.py`, does vocabulary frequency analysis against a 20k-word English corpus with no API cost.

**Consequences** — A second one-time API expenditure beyond the ~$6 summarization cost. Scenes enable the Journey timeline view and per-scene browsing. Vocabulary analysis is deterministic and free to rerun. Exposed as `make scenes` and `make vocab` (see ADR 4.0).

## ADR 3.1: Entity canonicalization as deterministic post-processing passes
**Date:** 2026-04-27
**Status:** Accepted

**Context** — ADR 2.0 deferred character deduplication and kept the raw `name` column as the join key, but variants ("Dr. Talos" vs "Doctor Talos", "autarch" vs "The Autarch") fragment cross-chapter joins. Raw location strings (1,550 of them) have the same problem.

**Decision** — Canonicalization runs as deterministic passes over the SQLite database after `make db`, not during extraction. `canonicalize_locations.py` normalizes 1,550 raw location strings into 172 canonical locations with hierarchy, type, and temporal annotations. `normalize_characters.py` merges case/prefix variants across the `characters` and `scene_characters` tables.

**Consequences** — Raw extracted data is preserved; canonicalization is reversible by re-running `make db`. Cross-chapter and multi-filter joins (browser additive pills) become reliable. Resolves the deduplication deferred in ADR 2.0. Exposed as `make canonicalize` and `make normalize`, ordered after `db` (see ADR 4.0).

## ADR 4.0: Full pipeline ordering in the Makefile
**Date:** 2026-05-18
**Status:** Accepted

**Context** — Scene decomposition (ADR 3.0) and entity canonicalization (ADR 3.1) were run manually with no `make` targets, so `make all` no longer reproduced the project's actual state. The ordering is non-obvious: `build_db.py` calls `DB_PATH.unlink()` and rebuilds the database from scratch, discarding any scenes, canonical locations, and name normalization applied afterward.

**Decision** — Add targets `scenes`, `canonicalize`, `normalize`, and `vocab`, and define `make all` as extract → summarize → db → scenes → canonicalize → normalize → browser. Constraints encoded in this order: scenes/canonicalize/normalize must follow `db` because it wipes the database; `normalize` follows `scenes` because it rewrites the `scene_characters` table; `browser` runs last because it joins the canonical-location tables. `vocab` is omitted from `all` — it is a local, API-free side analysis not consumed by the browser.

**Consequences** — `make all` reproduces the full current state. `extract_scenes.py` reuses `data/scenes.json` as a cache, so re-running `all` after a `db` rebuild re-inserts scenes without new API cost. The destructive `make db` step remains the reason the post-processing passes must rerun.

## ADR 5.0: Separate generic core from per-book study code
**Date:** 2026-05-18
**Status:** Accepted

**Context** — The pipeline was built for one book but most of it is not book-specific. Measured: `build_db.py` contains zero "Book of the New Sun" tokens; `summarize.py`, `extract_scenes.py`, and `analyze_vocab.py` contain only a handful, all in prompt flavor text. Only `extract.py` (the `b1/b2/b3` omnibus mapping) and `normalize_characters.py` (a hardcoded entity map) genuinely encode knowledge of this book. The risk runs both ways: keeping everything book-specific blocks reuse, but building a generic plugin framework after only one book means designing the abstraction blind.

**Decision** — Split the code by a single test: *would a second book reuse this file as-is, or rewrite it?*
- `core/` — reused as-is: `summarize.py`, `build_db.py`, `extract_scenes.py`, `analyze_vocab.py`, `build_browser.py`.
- `studies/botns/` — rewritten per book: `extract.py` (corpus ingest adapter), `canonicalize_locations.py` and `normalize_characters.py` (entity-resolution rules), `RESEARCH.md` (the research questions), `extract_deaths.py` (a study-specific extractor).

This *names the seam* without building a framework. A second book becomes a new `studies/<book>/` directory. No config system, no plugin loader, no parameterized prompts yet — those are deferred until a second book actually demonstrates what must vary. Scripts are relocated unchanged; they have no inter-script imports and resolve `data/` relative to the working directory, so `make` (run from the repo root) is unaffected beyond the path prefixes.

**Consequences** — The core is reusable; book knowledge is quarantined in one directory. The known imperfection: core scripts still carry "Gene Wolfe" flavor in their prompts — acceptable, and deliberately not abstracted until book #2 forces the question. Splitting `extract.py` into a generic epub parser plus a study config is also deferred for the same reason.

## ADR 5.1: Death events as a typed extraction, not a keyword search
**Date:** 2026-05-18
**Status:** Accepted

**Context** — Research question #3 ("deaths") was scoped only as the word "deaths" and never built. An ad-hoc free-text search across summaries and `factual_claims` missed the avern duel — a central near-death of the protagonist — because the text records only that "the leaf did not kill Severian." The data model has no notion of an *event*: a death is scattered across chapter-summary rows and unlinked claims. A keyword search also cannot represent a death that is prevented, reversed, or figurative.

**Decision** — Treat a death event as a first-class typed record extracted via the Claude API, by `studies/botns/extract_deaths.py` → the `death_events` table. The schema captures `subject`, `cause`, `agent`, `event_type`, `outcome` (`final` | `prevented` | `reversed` | `figurative` | `ambiguous`), `mechanism` (how death is averted or undone, when not final), `epistemic_tier` (`witnessed` | `claimed` | `figurative` | `inferred`), `evidence`, and `confidence`. The extraction prompt explicitly demands prevented, near, reversed, and figurative deaths — not only completed ones. The `epistemic_tier` field carries ADR 1.1's three-tier discipline down to the individual event, so an interpretive call (does the avern duel "count" as a death?) is tagged rather than asserted.

**Consequences** — RQ#3 becomes a SQL query over typed rows instead of a grep. The extraction mechanism is generic; the breadth-of-definition and the choice of book belong to the study, which is why the script lives in `studies/botns/`. Costs one API pass over 144 chapters; `data/deaths.json` caches it so reruns after `make db` are free. The browser has a Deaths view that shows an empty state until `make deaths` populates the table.

## ADR 6.0: Hypothesis workbench — deterministic retrieval, API scoring
**Date:** 2026-05-18
**Status:** Accepted

**Context** — A reader's theory about the text ("Dorcas is Severian's grandmother") is interpretive, not enumerable. Scoring it well needs a language model. But binding the whole feature to the API would make it dead weight whenever no key is available, and would hide where the rigour actually comes from.

**Decision** — `core/hypothesis.py` splits the feature into two layers. (1) **Retrieval** is deterministic: a SQLite **FTS5** index (`evidence_fts`) over every text field in the database — chapter prose, summaries, factual claims, notable details, scenes, embedded narratives. A hypothesis is reduced to content terms and matched with bm25 ranking. (2) **Scoring** is a single Claude API call that reads the hypothesis plus the retrieved evidence and returns a structured verdict, supporting and contradicting points with locators, and an epistemic caveat. Without `ANTHROPIC_API_KEY` the tool runs retrieval-only: it prints the evidence and skips scoring. Reports persist in a `hypotheses` table.

A hypothesis is tagged interpretive per ADR 1.1: the scoring prompt is instructed to weigh evidence and give a reasoned reading, never to claim proof, and to actively seek contradicting evidence.

**Consequences** — ~75% of the feature (index, retrieval, ranking, report format, persistence) works with no API and is fully testable offline. The API is one pluggable call. The `evidence_fts` index is trivially regenerable (~1s, no API), so it is **not** committed in `botns.db` — it auto-builds on first run and via `make index`. The `hypotheses` table lives in `botns.db` and is therefore cleared by the destructive `make db`; acceptable, since a hypothesis can simply be re-run.

## ADR 7.0: Per-fact `revealed_at`, not per-character identity axis
**Date:** 2026-05-23
**Status:** Accepted

**Context** — The Brown Book product depends on a "spoiler horizon": readers set a reading position (book + chapter) and the tool fogs anything past that point. Implementing this requires a `revealed_at` axis on every queryable thing. A naive per-character model — one field per entity indicating first appearance — leaks identity-level reveals: Dorcas exists from B1 Ch21 (mentioned-at), but the "she's Severian's grandmother" fact only appears in B3. A character-level field either reveals that fact too early (alongside her existence) or hides her entirely. A two-axis model on Character (mentioned-at + identity-revealed-at) requires curating which characters have hidden identities and writing it as a separate axis — workable but redundant.

**Decision** — Carry `revealed_at: BookChapter` on every fact-bearing row (`factual_claims`, `notable_details`, `scenes`, `embedded_narratives`, `death_events`), defaulted to the source chapter where the fact was extracted. Characters carry only `mentioned_at` (first appearance, auto-computed). The fact-level axis is correct by construction for the vast majority of cases: a fact extracted from B3 Ch11 has `revealed_at = (3, 11)`. Rare overrides (explicit foreshadowing whose `revealed_at` differs from its source chapter) are handled by the corrections layer (ADR 8.0).

**Consequences** — Spoiler correctness is structural, not visual: every query filters facts by `revealed_at ≤ position`. Characters themselves can appear (mentioned in early chapters) without leaking later identity facts. Population cost is trivial: a column with a default-from-source-chapter. Override cost lives in `corrections/seed/identity-reveals.yaml`, reviewable as data. Index on `(revealed_at_book, revealed_at_chapter)` keeps the filter cheap.

## ADR 8.0: Corrections as PR-driven YAML, applied at build + read time
**Date:** 2026-05-23
**Status:** Accepted

**Context** — LLM extractions are imperfect. The tool must let readers flag errors ("these two characters are the same"; "this fact is misattributed") without a backend, while keeping the durable record reviewable in the open. A correction needs to apply at *read time* (instant UI feedback once a fix is merged) and at *build time* (so subsequent pipeline runs converge on the fixed state, and stay correct under db rebuilds that ADR 4.0 mandates).

**Decision** — Corrections live as YAML files in a `corrections/` directory in the public repo (ADR 9.0). Each file is one correction, one of three v1 types: `merge` (two entities become one), `rename` (canonical_name change), `factual_fix` (edit a field on a claim/detail/scene/etc.). The schema is enforced by a Pydantic discriminated union (`Correction` in `core/models/corrections.py`). A new pipeline pass `core/apply_corrections.py` reads `corrections/*.yaml` and dispatches by type after `canonicalize` and before `scene-locations` / `normalize` / `metrics` / `index` / `browser`. Readers submit corrections via a flag affordance in the SPA that opens a pre-filled GitHub PR. Merged PRs trigger a CI rebuild + redeploy.

**Consequences** — GitHub is the audit log; no separate moderation surface. `factual_fix` corrections include an `old_value` safety check that rejects stale corrections after a db rebuild (protects against the underlying row having changed). Most correction types are API-free at build time (entity merges and field edits don't require re-extraction), so most CI runs cost zero in API spend. The hardcoded entity-merge map in `studies/botns/normalize_characters.py` becomes a candidate for migration into seed YAML — deferred as a v1.1 cleanup.

## ADR 9.0: Two-repo split — private workshop + public Brown Book
**Date:** 2026-05-23
**Status:** Accepted

**Context** — `analysis-botns` contains the full extraction pipeline (`source.epub`, API-dependent scripts, working notes, the `JOURNAL.md` of decisions and mistakes). Shipping a public tool from the same repo would mix Elliott's personal workshop with the contributor-facing surface. Contributors don't need (and arguably shouldn't see) the messy parts; the public surface should be small, clean, and focused on the SPA + corrections workflow.

**Decision** — Split into two repos. `analysis-botns` (private) owns the extraction pipeline, source materials, and full history. `the-brown-book` (public) owns the SPA (`web/`), the live `corrections/` directory, and vendored copies of `core/models/`, `core/apply_corrections.py`, `core/horizon.py`, plus the vendored `data/botns.db`. A `make publish` target in analysis-botns syncs the canonical artifacts to the public repo via local filesystem (`PUBLIC_REPO ?= ../the-brown-book`). Most CI activity (community correction PRs) runs in `the-brown-book` and never needs the Anthropic API key. New extraction work (rare after v1) runs in `analysis-botns` and pushes via `make publish`.

**Consequences** — Two repos to maintain. Drift risk between vendored copies is mitigated by `make publish` being the only refresh path. Corrections live in only one place (public). Contributors interact with a small, clean surface. The cost is a manual sync step on the rare occasion that extraction work happens; for v1 this is acceptable. A CI-driven cross-repo sync could land in v1.1 if the manual step proves friction.

## ADR 10.0: Browser-side SQLite via sql.js
**Date:** 2026-05-23
**Status:** Accepted

**Context** — The pipeline produces a ~9MB SQLite database with an FTS5 evidence index over every text field. The product needs deterministic retrieval at query time with no backend and no per-query API costs. Three options: (a) ship the SQLite file to the client and query it in-browser; (b) pre-compute every panel result to JSON at build time; (c) run a query server. (c) violates the no-backend principle (ADR 9.0 and the PRD's "trust without a backend" framing); (b) requires anticipating every possible question and rebuilding the artifact for every new panel; (a) preserves the deterministic retrieval semantics of the existing pipeline.

**Decision** — Ship `data/botns.db` as a static asset. Load it in the browser via **sql.js** (SQLite compiled to WebAssembly, FTS5 enabled). All panel queries run client-side via prepared statements; SQL kept as `.sql` files imported via Vite's `?raw` suffix. DuckDB-WASM was considered for faster aggregations but rejected — the FTS5 index lives in SQLite and migrating it would be a major project for marginal benefit at this corpus size.

**Consequences** — First-load weight is real: ~2.5MB DB (Brotli-compressed) + ~700KB WASM + bundle. Mitigated by content-hashed filenames + far-future caching, plus loading in the background during Scene 1's reading-position picker (so the user never sees a spinner on first visit). Adding new panels in the future requires no build-pipeline change — the DB has everything; the SPA just adds a new query. Hand-mirrored TS types in `web/src/lib/types.ts` must stay in sync with the Pydantic models in `core/models/`; an `assertSchema()` check at boot catches drift before it surfaces as cryptic SQL errors.

## ADR 11.0: Frontend stack — Svelte + Vite + Cloudflare Pages
**Date:** 2026-05-23
**Status:** Accepted

**Context** — The Brown Book is a single-page tool with three routes (Entry, Workshop, About), reactive panel updates driven by theory + position, and no server. SvelteKit, Next.js, Remix all add server-side rendering and file-based routing that this product doesn't need. Vanilla JS + a small templating helper would work but trades developer ergonomics for marginal bundle savings. Hosting on GitHub Pages is simpler (one less account) but lacks Brotli compression for arbitrary file types and built-in PR preview deploys.

**Decision** — Plain Svelte 4 + Vite + TypeScript. Cloudflare Pages with its GitHub integration for deploys + PR previews. No SvelteKit (no SSR needed; three routes don't justify file-based routing). No service worker in v1 (the whole app is static, so offline support via SW is a v1.1 nice-to-have).

**Consequences** — Small bundle (~50KB Svelte runtime + app code), reactive panel reflow is idiomatic, no SSR overhead. Cloudflare's Brotli shrinks the DB asset meaningfully on first load. PR previews give correction reviewers a live URL per PR — they can see the merged result before approving. Custom domain remains a v1.1 concern; `<project>.pages.dev` is fine for first ship. The hand-mirroring of Pydantic → TS types (called out in ADR 10.0) is the main maintenance tax — accepted because it's small and `assertSchema()` catches drift.

## ADR 11.1: Look-ahead reveals all fogged passages inline, no per-item modal
**Date:** 2026-05-23
**Status:** Accepted

**Context** — TASK-022 required a decision on how "Look ahead" works in PassagesPanel: (a) a per-item confirmation modal (DESIGN.md Scene 5 sketched this) or (b) batch inline expansion of all fogged items with a single click.

**Decision** — Inline batch expansion. Clicking "Look ahead →" in the fog footer expands all past-horizon items that are already loaded in the panel, immediately records them all to `bb:v1:revealed` in localStorage, and renders them with the permanent ↪ tag. No modal, no per-item confirmation.

**Consequences** — Simpler code and lower click cost. The trade-off is that users cannot peek at one item without revealing all loaded fogged items. This is acceptable for v1: the panel shows at most 50 results (topN), and users who look ahead generally want the full picture. A per-item reveal with individual confirm modals can be added in v1.1 if there is reader demand. The batch behavior is honest: the ↪ tag appears on every revealed item so readers know exactly what they peeked at.
